Knots MCP Server (kno-mcp)

A buildable spec: implementation steps, design decisions, and — for every unit of work and every phase — acceptance criteria paired with reproducible, binary-outcome (go/no-go) verification commands. The verification steps are the rubric.

Grounded in the live codebase (file:line cited inline) and the official Rust MCP SDK (rmcp).

How to read this document
  • Architecture (§1–§3) — the decisions and the "why," condensed. Read once.
  • Global design decisions (§4) — cross-cutting choices (D1–D12) every unit assumes.
  • Verification harness (§5) — the reusable fixtures the go/no-go checks call.

1 · Architecture in one screen

The premise that reframes everything: Knots has no authoritative host. State is event-sourced JSON synced through a git branch (refs/work/knots); every machine with the remote holds a full replica (src/sync, src/replication.rs, src/sync_ref.rs). There is no server, daemon, or socket anywhere in the tree. Therefore an MCP server is not a gateway to an authority — it is just another replica with a network door, and the git remote is the only rendezvous.

What we build

A second binary kno-mcp in this crate. rmcp handles the MCP protocol + transport; our code maps tools → kno … --json invocations against a local replica, and runs a background git-sync loop.

Where it runs

On Manhattan (always-on), against a dedicated git checkout it owns. It holds the sync credential; sandboxes never do. A stdio variant also runs locally on the MacBook for on-box agents.

How sandboxes reach it

Streamable HTTP over a Tailscale tailnet + bearer token. The "which host?" question is one line of client config (a URL), never an architectural choice.

4 · Global design decisions

These are settled choices; units below assume them. Each states the decision, the rationale, and the rejected alternative.

#DecisionRationale / rejected alternative
D1Execute Knots by shelling out to the kno binary with --json (Option A), not by linking the crate in-process.Reuses the file-locked write queue, sync, and lease logic verbatim — cannot drift from CLI truth. Rejected: in-process KnotsApi (needs lib.rs carve-out + dispatch reuse) — deferred to Phase 3 only if latency is measured to matter.
D2Single language, single crate. kno-mcp is a second binary target in this repo using rmcp.One cargo build, one version, no skew. Rejected: TypeScript sidecar (second runtime + release cadence).
D3MCP deps are feature-gated. rmcp/tokio/axum are optional=true behind feature mcp; the bin declares required-features=["mcp"].Keeps the default knots binary lean; default builds don't pull async stack. Rejected: unconditional deps (bloats the shipped CLI).
D4Resolve the sibling kno by default (current_exe().parent()/"kno"); override with --kno-bin <path>.Guarantees the MCP server and CLI are the same release; makes tests hermetic (point at a stub). Rejected: bare kno on PATH (version ambiguity).
D5Store is selected by -C <repo> against a dedicated git checkout the service owns; every kno invocation passes it.Matches real store discovery (§3); the checkout has the sync remote. Rejected: a fictional KNOTS_HOME env (does not exist).
D6One lease per MCP session. On initialize, create a lease tagged with the client's AgentInfo; cache session→lease_id; knots_claim passes --lease.Gives attribution and honours claim semantics. Cleanup via timeout_seconds (+ explicit release if V0.5 confirms one exists).
D7Pass the boundary contract through untouched. knots_claim surfaces workflow_boundary_kind and e2e exactly as kno emits them; --e2e only when the caller sets it.Honours CLAUDE.md Claim-Boundary Precedence; never infers intent from prose.
D8Tool results carry both forms: the parsed JSON as structuredContent and pretty JSON as a text block.Structured for programmatic clients, text for models that read content. Cheap.
D9A failed kno call → a tool result with isError=true carrying the stderr text + exit code, not a transport-level JSON-RPC error.The agent sees and can react to domain errors; protocol stays healthy. (Knots has no JSON error envelope, §3.)
D10Thin I/O shell, pure testable core. argv-construction, JSON mapping, session/lease state, and auth checks are pure functions with unit tests; network bind, process spawn, stdio loop are glue marked #[cfg(not(tarpaulin_include))].Lets us hit the 95% gate (§3) without unit-testing real sockets. Uses the crate's existing tarpaulin-exclusion pattern.
D11Transport is staged: stdio in Phase 1 (locally testable, no network), Streamable HTTP in Phase 2.De-risks the protocol/tool layer before adding network + auth + deployment.
D12Auth = static bearer token (from a file/env), checked by axum middleware; Tailscale ACL is the second gate. OAuth 2.1 is a Phase 3 option.Adequate for a personal homelab; simplest correct thing. Rejected for now: full OAuth (overkill, more moving parts).

7 · Phase 1 — Core server over stdio

Goal: a fully functional Knots MCP server reachable over stdio, with the core tool set, per-session leasing, and the boundary contract — all unit-tested to the coverage gate. No network yet (D11).

U1.1KnoRunner — argv construction & invocation

Steps

  1. Pure fn build_argv(repo, subcommand, args) -> Vec<String> that always prefixes -C <repo> and appends --json (D5).
  2. Runner that spawns --kno-bin (D4) with that argv, captures stdout/stderr/exit.
  3. Map: exit 0 → Ok(serde_json::Value) (parse stdout); exit≠0 → Err{stderr, code} (D9).
Acceptance: argv is constructed deterministically; success parses JSON; failure surfaces stderr + code. build_argv and the mapping are pure and unit-tested.
V1.1a unit test: build_argv("/r","ls",["--state","ready"]) == ["-C","/r","ls","--state","ready","--json"] → GO test passes.
V1.1b integration with Fixture B: a list call returns data[0].id=="k1" → GO.
V1.1c integration with sentinel MISSING: returns isError with stderr containing not found, exit code 1 captured → GO.
U1.2Read tools: knots_list, knots_show, knots_poll

Steps

  1. knots_list(state?,tag?,type?,limit?) → ls with mapped flags.
  2. knots_show(id) → show <id>. knots_poll() → poll.
  3. Each returns structuredContent + text (D8).
Acceptance: each tool maps params → correct argv and returns the command's JSON unaltered; knots_show on a bad id is a non-fatal tool error (D9).
V1.2a Fixture A: seed 2 knots, call knots_list → structuredContent .total==2 and .data|length==2 → GO.
V1.2b knots_show on a seeded id → .id equals it and .state present → GO.
V1.2c knots_show("MISSING") → result has isError==true, text contains not found → GO.
U1.3Write tools: knots_create, knots_update

Steps

  1. knots_create(title, description?, acceptance?, type?, priority?) → new/q (uses U0.4's confirmed JSON).
  2. knots_update(id, fields…) → update.
Acceptance: creating returns the new id; updating mutates the named fields and the change is observable on re-read.
V1.3a Fixture A: knots_create("spec demo") → returns id $K; then kno -C $REPO show $K -j | jq -e '.title=="spec demo"' → GO.
V1.3b knots_update($K, priority=1) then show → .priority==1 → GO.
U1.4Workflow tools: knots_claim, knots_next, knots_rollback

Steps

  1. knots_claim(id, e2e?:bool) → claim <id> --lease <session_lease> [--e2e] (D6, D7). Pass workflow_boundary_kind and e2e straight through (D7).
  2. knots_next(id) → next <id>. knots_rollback(id) → rollback <id>.
Acceptance: a default claim yields workflow_boundary_kind=="single_action" and e2e==false; an e2e:true claim yields "e2e_continuation" and e2e==true; the values are byte-identical to what kno emits.
V1.4a Fixture A: claim a ready knot (no e2e) → structuredContent .workflow_boundary_kind=="single_action" and .e2e==false → GO.
V1.4b claim with e2e:true → .workflow_boundary_kind=="e2e_continuation" and .e2e==true → GO.
V1.4c diff: tool's structuredContent equals kno -C $REPO claim <id> --lease L --json output (passthrough fidelity) → GO if jq -S equal.
U1.5knots_sync + SyncOutcome handling

Steps

  1. knots_sync() → sync; surface the tagged enum.
  2. Recognize {"status":"deferred","active_leases":N} and report it as a distinct, non-error outcome.
Acceptance: a completed sync returns status=="completed" with push/pull summaries; a deferred sync returns status=="deferred" and is not surfaced as isError.
V1.5a Fixture A with a configured remote: knots_sync → .status=="completed" → GO.
V1.5b stub returns deferred → tool result isError==false and structuredContent .status=="deferred" → GO.
U1.6Per-session lease lifecycle

Steps

  1. On MCP initialize, read client info; run lease create --nickname mcp-<sid> --type agent --agent-name <n> --provider <p> --model <m> --model-version <v> --timeout-seconds <T> -j; store sid→lease_id (the returned .id).
  2. knots_claim binds that lease (U1.4).
  3. On session close: explicit release if U0.5 found one, else rely on timeout.

Decisions

D6. The session→lease map is a pure state machine, fully unit-tested (D10).

Acceptance: exactly one lease is created per session; claims within the session reuse it; the lease carries the caller's AgentInfo.
V1.6a Fixture A: open session, issue 2 claims, then kno -C $REPO ls --type lease -j | jq -e '[.data[]|select(.title|startswith("mcp-"))]|length==1' → GO (one lease, not two).
V1.6b created lease's agent_info.model equals the value passed at initialize → GO.
V1.6c unit test of the session map: second claim in a session does not call lease create → GO.
U1.7stdio transport & MCP handshake

Steps

  1. Wire KnoMcp to the stdio transport; main reads --kno-bin, -C/--repo. Mark the run loop #[cfg(not(tarpaulin_include))] (D10).
Acceptance: a real MCP client completes initialize and lists the tools from U1.2–U1.5.
V1.7a claude mcp add kno-stdio -- kno-mcp --repo $REPO then claude mcp list shows it connected → GO.
V1.7b a tools/list call returns ≥7 tools including knots_list, knots_claim, knots_next, knots_sync → GO.
V1.7c (CI-friendly) pipe a JSON-RPC initialize then tools/list to kno-mcp stdin; assert the responses contain serverInfo and the tool names → GO.

Phase 1 exit gate

  • Functional: V1.1–V1.7 all GO.
  • End-to-end claim loop (real binary): against Fixture A, a script drives initialize → knots_list → knots_claim → knots_next and the knot's state changes between claim and next. GO if the post-next state ≠ the pre-claim state.
  • Quality: make sanity GO (coverage ≥95% with the pure core fully tested; glue excluded per D10).

8 · Phase 2 — Remote & always-on

Goal: the same server, reachable by a sandbox agent over the network, running unattended on Manhattan, kept fresh by a background sync loop. This is the actual product.

U2.1Streamable HTTP transport

Steps

  1. Enable rmcp feature transport-streamable-http-server; mount the service on an axum router at /mcp; bind --bind <addr:port> (default 127.0.0.1:7777). Glue is #[cfg(not(tarpaulin_include))] (D10).
Acceptance: an MCP initialize over HTTP returns serverInfo; tools are listable over HTTP exactly as over stdio.
V2.1a start bound to localhost; curl -s -XPOST localhost:7777/mcp -d '{...initialize...}' → response JSON contains "serverInfo" → GO.
V2.1b claude mcp add --transport http knots http://localhost:7777/mcp + tools/list shows the same ≥7 tools as V1.7b → GO.
U2.2Bearer-token auth middleware

Steps

  1. Load token from --token-file/env (D12).
  2. axum middleware: reject requests whose Authorization: Bearer ≠ token with 401; the token-compare fn is pure + unit-tested (constant-time compare).
Acceptance: no/incorrect token → 401 and the tool does not execute; correct token → 200 and normal operation.
V2.2a curl -s -o /dev/null -w '%{http_code}' -XPOST localhost:7777/mcp -d '{...}' (no header) → prints 401 → GO.
V2.2b wrong token → 401 → GO.
V2.2c correct token → 200 and an initialize result → GO.
V2.2d unit test: compare fn rejects empty/short/mismatched tokens → GO.
U2.3Background sync loop + freshness

Steps

  1. Spawn a tokio task running kno -C repo sync every --sync-interval (default 15s); also sync immediately after any mutating tool (claim/next/create/update).
  2. On SyncOutcome::Deferred, log and retry with backoff (CLI already retries internally — U1.5 surfaces it).
Acceptance: a change written via the MCP server appears on a second replica within one sync interval; a deferred sync is retried rather than crashing the loop.
V2.3a two checkouts of the same remote (R1 served by kno-mcp, R2 plain). Create a knot via the MCP tool; wait one interval; kno -C R2 sync && kno -C R2 ls -j | jq -e '[.data[]|select(.title=="…")]|length==1' → GO.
V2.3b inject a deferred outcome (stub) → the loop logs a retry and keeps running (process still alive after 2 intervals) → GO.
U2.4Manhattan deployment (service + dedicated checkout)

Steps

  1. Provision a git checkout at /var/lib/knots/repo with origin = the Knots remote and the sync credential available to the service user only.
  2. Install a systemd unit (Linux) / launchd plist (macOS) running kno-mcp serve --repo /var/lib/knots/repo --bind <tailnet-ip>:7777 --token-file … --sync-interval 15s, Restart=always.
Acceptance: the service starts on boot, restarts on crash, and answers MCP requests; the git credential is not present in any sandbox.
V2.4a systemctl is-active kno-mcp (or launchctl print) → active/running → GO.
V2.4b kill the process; within 5s is-active is again active → GO.
V2.4c from the service host, an authenticated initialize returns serverInfo → GO.
U2.5Tailscale exposure & isolation

Steps

  1. Bind to the tailnet interface (or front with Tailscale Serve for TLS).
  2. Set a tailnet ACL allowing only sandbox nodes to reach :7777.
Acceptance: a node on the tailnet (with token) can reach the endpoint; a host off the tailnet cannot.
V2.5a from a second tailnet node: authenticated initialize to https://manhattan.<tailnet>.ts.net/mcp → serverInfo → GO.
V2.5b from a non-tailnet host: curl --max-time 5 to the same name/IP → connection refused/timeout (non-zero curl exit) → GO.
U2.6End-to-end: sandbox agent claims through MCP

Steps

  1. Configure a sandboxed Claude Code with the remote MCP server (URL + token).
  2. Have it claim a knot and advance it; observe the change land on the MacBook after sync.
Acceptance: a sandbox with only a URL+token (no git, no kno) can list, claim (under an attributed lease), and advance a knot, and the result is visible on the MacBook replica.
V2.6a sandbox runs knots_claim on a ready knot → returns a single_action boundary; the knot's lease on the shared store carries the sandbox's agent identity → GO.
V2.6b sandbox runs knots_next; after a sync, kno ls on the MacBook shows the advanced state → GO.

Phase 2 exit gate

V2.1–V2.6 all GO, plus make sanity GO. At this point the product's core promise is demonstrably met: a sandbox co-worker drives Knots with no local state and no credentials, attributed and boundary-respecting, converging through git.

rmcp tool shape (per the SDK macros)

use rmcp::{ServerHandler, handler::server::wrapper::Parameters,
           model::*, tool, tool_handler, tool_router, ErrorData as McpError};

#[derive(serde::Deserialize, schemars::JsonSchema)]
struct ListArgs { state: Option<String>, tag: Option<String>, limit: Option<u32> }

#[tool_router]
